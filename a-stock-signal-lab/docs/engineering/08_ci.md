# 08 GitHub Actions Quality Gates

## 表结构

本步不改变业务数据库。CI 只在临时 SQLite 中运行全部可逆 migrations、固定决策回放和基准回归；可回滚边界是 workflow、三个 CI 脚本和两个 fixture 的独立提交。

## 接口定义

- `python -m ruff check fenjue scripts tests --select E9,F63,F7,F82`：语法、未定义名称和致命 Python lint。
- `python -m unittest discover -s tests -v`：全部单元与契约测试。
- `python scripts/ci_migration_dry_run.py`：从完整库执行全 migration down/up 并校验版本和完整性。
- `python scripts/ci_fixture_replay.py`：同一固定上下文回放 research/shadow/production，校验 graph hash 和执行权限。
- `python scripts/ci_shadow_baseline_regression.py`：固定机会集校验 shadow 相对 baseline 的 lift、净期望、Brier 和切片覆盖。

GitHub workflow 使用官方 `actions/checkout@v6` 与 `actions/setup-python@v6`，五项门分别为独立 job，避免一个失败掩盖其他结果。

## 伪代码

```text
on push or pull request touching project:
  run lint
  run unit tests
  build temporary DB and replay every migration down/up
  replay deterministic three-mode decision fixture
  compare deterministic shadow sample against baseline thresholds
  require all five jobs green
```

## 回滚方案

删除 `.github/workflows/quality.yml` 即停止远端门禁；回退本步提交可同时移除 CI 脚本和 fixture，不影响运行时表或 Hermes 数据。若仅某个 fixture 口径升级，应保留旧 fixture 历史并在新提交中解释阈值变化，禁止静默降低门槛。

## 测试样例

1. CI 契约测试确认 workflow 含五个 job 和对应命令。
2. 决策 fixture 明确 research 不可执行、production 可执行以及共同 graph action。
3. shadow/baseline fixture 至少覆盖两个交易日、两个逻辑簇和六个机会。
4. 本地逐条运行与 Actions 完全相同的五条命令。
